# Урок 4: Продвинутый Source-режим: синхронизация, хуки и индикаторы (v4.33)

**Цель урока:** Освоить профессиональные инструменты `Source Builder`: автоматическую синхронизацию настроек, систему контроля ресурсов, использование хуков для патчинга ядра и механизмы импорта бинарных пакетов (Binary-to-Source).

---

### 📊 1. Мониторинг ресурсов (Система индикаторов)

В версии **4.33** панель мониторинга в главном меню приняла вид **`[F P S M H X | OI OS]`**. Это ваш "пульт управления", позволяющий мгновенно оценить состав профиля перед сборкой:

*   **F (Files)**: Указывает на наличие папки оверлея `custom_files/%ID%`. Здесь хранятся ваши конфиги, скрипты и файлы, которые попадут в корень роутера.
*   **P (Packages)**: В папке `custom_packages/%ID%` найдены готовые `.ipk` пакеты для импорта.
*   **S (Source)**: В папке `src_packages/%ID%` находятся исходные коды сторонних пакетов (склонированные из Git или созданные импортером).
*   **M (Manual Config)**: Сигнализирует о наличии активного файла `manual_config`. Это значит, что у вас есть сохраненный результат сессии Menuconfig, который еще не перенесен в профиль.
*   **H (Hooks)**: В папке оверлея найден скрипт автоматизации `hooks.sh`.
*   **X (Patches)**: В папке `custom_patches/%ID%` есть файлы патчей для наложения на исходный код OpenWrt.
*   **OI / OS**: Индикаторы готовой продукции (**Image / Source Builder** соответственно).

### 🔄 2. Новая синхронизация настроек (Auto-Sync)

В версии **4.33** реализован бесшовный цикл синхронизации:

1.  **Запуск**: Вы выбираете пункт **[K] Menuconfig**.
2.  **Настройка**: Вносите изменения в графическом интерфейсе OpenWrt.
3.  **Выход**: После закрытия контейнера Билдер выводит запрос: **"Обновить профиль данными из Menuconfig? [Y/N]"**.
4.  **Синхронизация**: При выборе **Y** настройки автоматически записываются в ваш файл профиля `profiles/%ID%.conf` в переменную `SRC_EXTRA_CONFIG`. Запись ведется в многострочном формате, поддерживающем комментарии `# ... is not set`.

**Результат:** Ваш `.conf` файл становится "единственным источником истины". Оригинальный конфиг архивируется в `applied_config_*.bak`.

### 📦 3. Импорт бинарных пакетов (Механизм Binary-to-Source)

Если у вас есть программа в виде `.ipk` пакета, но нет её исходников (закрытое ПО, драйверы), Билдер позволяет «обернуть» её в полноценный пакет для Source Builder.

#### А. Алгоритм работы:
1.  **Размещение**: Поместите `.ipk` файлы в папку `custom_packages/%ID%/`.
2.  **Запуск**: В главном меню выберите опцию **[I] (Импорт IPK)**.
3.  **Автоматическая обработка**: PowerShell-скрипт `system/import_ipk.ps1` выполнит:
    *   **Распаковку**: Извлечет метаданные (`control`) и данные (`data.tar.gz`).
    *   **Проверку архитектуры**: Сравнит архитектуру пакета с целевой архитектурой вашего профиля.
    *   **Коррекцию зависимостей**: Автоматически заменит устаревшие зависимости на актуальные для OpenWrt 23.xx/24.xx.
    *   **Генерация Makefile**: Создаст в `src_packages/%ID%/%имя_пакета%/` специальный Makefile.

#### Б. Что происходит "под капотом" в сгенерированном Makefile:
*   **Сохранение структуры**: Архив `data.tar.gz` не распаковывается на Windows, а копируется как есть. Распаковка происходит внутри Linux-контейнера. Это **сохраняет символические ссылки (symlinks)** и корректные права доступа.
*   **Защита бинарных данных**: Команды `STRIP:=:` и `PATCHELF:=:` запрещают системе сборки изменять (оптимизировать) бинарные файлы.
*   **Безопасные скрипты**: Установочные скрипты (`postinst`) оборачиваются в проверку `if [ -z "$$IPKG_INSTROOT" ]`, чтобы они не запускались во время компиляции, а срабатывали только при установке на роутер.

### 🪝 4. Автоматизация через Hooks и Vermagic Hack

Скрипт `custom_files/%ID%/hooks.sh` выполняется внутри контейнера перед компиляцией и является мощным инструментом для глубокой модификации сборочного процесса. В шаблоне `scripts/hooks.sh` уже заложены полезные функции:

*   **Авто-включение Wi-Fi:** При использовании хука в образ автоматически добавляется `uci-defaults` скрипт. Он сработает один раз при первой загрузке роутера и включит все беспроводные интерфейсы, избавляя от необходимости подключаться по кабелю для начальной настройки.
*   **Добавление фидов (внешних репозиториев):** В `hooks.sh` есть функция `add_feed`. Вы можете раскомментировать или добавить новые строки для подключения целых репозиториев с пакетами, которых нет в OpenWrt. Пример из скрипта:
    ```bash
    # Добавляет репозиторий с пакетом amneziawg
    add_feed "amneziawg" "https://github.com/amnezia-vpn/amneziawg-openwrt.git"
    ```
*   **Патчинг**: Используется для наложения патчей (`patch -p1`) на исходный код.
*   **Vermagic Hack (Автоматически)**: Эта проблема решается в Билдере автоматически для релизных сборок. Скрипт `hooks.sh` сам определяет вашу версию OpenWrt, скачивает официальную "подпись" ядра (`vermagic`) и встраивает ее в вашу сборку. Это позволяет устанавливать `kmod-` пакеты из официальных репозиториев без ошибок совместимости.

#### 🛡️ Архитектура надежности: Механизм «самовосстановления» (v4.09+)

Одна из самых мощных, но не очевидных функций сборщика — защита от "грязного" состояния среды сборки.

*   **Проблема:** Использование `hooks.sh` для применения патчей (например, Vermagic Hack) оставляет след в рабочей директории. Если после этого вы решите собрать прошивку для другого профиля (или для этого же, но уже без `hooks.sh`), остаточные изменения от патчей могут вызвать трудноуловимые ошибки компиляции. Вы получите "грязную" сборку, даже не подозревая об этом.

*   **Решение — Автоматический откат:** Начиная с версии 4.09, сборщик перед каждым запуском в режиме `Source` проводит проверку. Если он обнаруживает, что `hooks.sh` в текущем профиле **отсутствует**, но среда сборки **была изменена** ранее, он запускает **полный цикл самовосстановления**:
    1.  Восстанавливает оригинальные файлы `Makefile` из резервных копий.
    2.  Удаляет маркеры `vermagic`.
    3.  Выполняет глубокую очистку кэша ядра (`make target/linux/clean`).
    4.  **Полностью сбрасывает CCACHE** для данной рабочей директории.

*   **Результат:** Этот механизм гарантирует, что каждая сборка без хуков начинается в **предсказуемо чистом состоянии**. Это значительно повышает надежность и воспроизводимость сборок, избавляя вас от необходимости вручную чистить кэши.

### 🎛️ 5. Управление ресурсами сборки (SRC_CORES)

Вы можете управлять нагрузкой на систему через переменную `SRC_CORES` в `.conf` файле:
*   **`SRC_CORES="safe"`**: Использует `N-1` ядер. Сборка идет быстро, но система остается отзывчивой.
*   **Не указано**: Использует **все** доступные потоки процессора.
*   **`SRC_CORES="4"`**: Жесткое ограничение количества потоков (например, для предотвращения перегрева ноутбука).
*   **`SRC_CORES="debug"`**: Принудительный запуск в один поток с подробным выводом (`make -j1 V=s`), необходимый для диагностики ошибок сборки.

### 🧹 6. Управление кэшем и решение проблем (CLEAN)

В главном меню **[C] CLEAN / MAINTENANCE** доступны следующие опции:

*   **[1] SOFT CLEAN (make clean):** "Первая помощь". Удаляет скомпилированные файлы для выбранного профиля, но оставляет исходники и тулчейн. Идеально, если сборка упала на середине.
*   **[2] HARD RESET (Удалить src-workdir):** Удаляет всю папку с исходниками для профиля. Используйте, если исходный код повредился или нужно начать "с чистого листа". Загруженные архивы (`dl`) и `ccache` сохранятся.
*   **[3] Очистить кэш исходников (dl):** Удаляет все скачанные архивы пакетов. Полезно для освобождения места. При следующей сборке они будут скачаны заново.
*   **[4] Очистить CCACHE:** Полностью сбрасывает кэш компилятора. Следующая сборка будет такой же долгой, как и самая первая. Используйте, если подозреваете проблемы с компилятором.
*   **[5] Очистить tmp (Package Index):** Удаляет служебную папку `tmp/` в дереве исходников (индекс пакетов). Помогает при сбоях обновления зависимостей или после смены фидов.
*   **[6] FULL FACTORY RESET:** "Ядерный вариант". Удаляет для профиля **всё**: исходники, кэш `dl` и `ccache`.
*   **[9] Prune Docker:** Глобальная команда Docker для очистки всей системы от неиспользуемых образов, сетей и кэшей. Полезна для общего обслуживания системы.

### 📦 7. Локальный Image Builder после Source (v4.40+)

После успешной сборки из исходников (режим **Source Builder**) билдер ищет в каталоге вывода архив Image Builder (`*imagebuilder*.tar.zst`) и предлагает обновить профиль — записать путь к этому архиву в `IMAGEBUILDER_URL`. Если согласиться (**Y**), следующие сборки можно делать уже в режиме **Image Builder** за 1–2 минуты, без повторной компиляции ядра.

**Шаги:** собрали прошивку из исходников → в конце сборки появился запрос «Обновить профиль? [y/N]» → **Y** → переключитесь в режим **M** (Image Builder) → выбирайте профиль и собирайте. Дальнейшие изменения пакетов и конфигов делаются через быстрый IB, пока не понадобится снова менять ядро или defconfig.

### 💡 8. Лайфхак: "Холодный старт" (на примере Rax3000M)

**Сценарий «один раз Source — потом IB за 1–2 минуты»:** не набивайте в Source Builder все пакеты под завязку. Встройте в menuconfig только нужные `kmod-` и минимум необходимого; в конце сборки включите сборку Image Builder (в menuconfig внизу), согласитесь обновить профиль локальным IB, переключитесь в режим Image Builder (**M**) и дальше собирайте прошивки с любым набором софта и конфигов за 1–2 минуты. Повторная полная сборка из исходников — от 10 минут с кэшем и до 30+ при первой.

Если вам нужно собрать прошивку по чужому сложному `defconfig`:
1.  Запустите **Menuconfig**, чтобы создались папки профиля, и выходите.
2.  Положите чужой файл конфигурации по адресу `firmware_output/sourcebuilder/%ID%/manual_config`.
3.  Снова запустите **Menuconfig**. Система обнаружит файл, расширит его и докачает зависимости.
4.  При выходе согласитесь на обновление профиля (**Y**). Весь сложный конфиг теперь интегрирован в ваш `.conf`.

### 💡 Продвинутый пример: Сборка с кастомными драйверами (Rax3000M)
Ниже описан пошаговый процесс "холодного старта" такой сборки:

1.  **Шаг 1: Настройка профиля (`.conf`)**
    - В файле профиля для `Source Builder` точно укажите URL репозитория (`SRC_REPO`) и целевую ветку (`SRC_BRANCH`), содержащую нужные драйверы.

2.  **Шаг 2: Создание структуры (`menuconfig`)**
    - Запустите `[K] MENUCONFIG` для этого профиля. Поскольку кэш и рабочая директория пусты, система создаст все необходимые папки, включая `firmware_output/sourcebuilder/<имя_профиля>/`.

3.  **Шаг 3: Поиск базового `defconfig`**
    - В репозитории исходного кода, который вы собираете, найдите базовый файл конфигурации. Обычно он лежит в папке `defconfig` и имеет минимальный размер (10-15 КБ).
    - *Пример:* `https://github.com/padavanonly/immortalwrt-mt798x-6.6/blob/openwrt-24.10-6.6/defconfig/mt7981-ax3000.config`

4.  **Шаг 4: Подмена конфигурации**
    - Скопируйте найденный `defconfig` в папку вывода и переименуйте его в `manual_config`.
    - **Путь:** `firmware_output\sourcebuilder\<имя_профиля>\manual_config`

5.  **Шаг 5: Разрешение зависимостей**
    - Снова запустите `[K] MENUCONFIG` для этого же профиля.
    - Система сборки OpenWrt обнаружит минималистичный `manual_config` и **автоматически расширит его**, включив все необходимые зависимости и опции. Ваш файл из 11 КБ превратится в полноценный `.config` размером ~400 КБ после чего будет опять ужат до 12кб и сохранён как manual_config. Если вы согласитесь в внесением настроек в профиль то файл будет переименован.


   * package/<имя_пакета>/compile: Это цель make, которая указывает системе сборки OpenWrt скомпилировать конкретный пакет. Замените <имя_пакета> на фактическое имя директории пакета в openwrt/buildroot/package/.
   * V=s: Этот флаг делает вывод сборки более подробным (verbose), что очень полезно для отладки, если что-то пойдет не так.

  Пример:

  Если вы хотите пересобрать пакет luci-app-samba4, команда будет такой:


   make package/luci-app-samba4/compile V=s

  Если вы просто хотите собрать пакет, а не пересобирать, и он ещё не был собран, можно использовать:

   make package/<имя_пакета>/install V=s
