# Урок 8: Встраивание APK/IPK пакетов в Image Builder

**Цель:** Научиться добавлять кастомные пакеты в режим Image Builder (IB) — от простого копирования до автоматической валидации и исправления имён файлов.

> **Кому это нужно:** если вы хотите поставить сторонний `.apk`/`.ipk` (LuCI-тему, плагин, драйвер) в свою прошивку, но не хотите собирать из исходников.

---

## 1. Два режима билдера и где живут пакеты

| Режим | Папка пакетов | Что делает билдер |
|-------|--------------|-------------------|
| **Image Builder (IB)** | `custom_packages/` | Копирует `.apk` напрямую в контейнер IB — они ставятся как готовые бинарники |
| **Source Builder (SB)** | `src_packages/` | Извлекает метаданные, генерирует Makefile и собирает из исходников |

**Этот урок посвящён режиму Image Builder** — самому быстрому способу получить прошивку с дополнительными пакетами.

---

## 2. Базовый метод: «положил и собрал»

Самый простой способ — скопировать `.apk` файлы в папку `custom_packages/`:

```
custom_packages/
└── <PROFILE_ID>/
    ├── luci-app-samba4.apk
    ├── luci-theme-argon.apk
    └── my-custom-package.apk
```

> **Где взять PROFILE_ID?** Это имя вашего профиля без `.conf`. Например, если профиль называется `giga_24105_main_full.conf` — папка будет `custom_packages/giga_24105_main_full/`.

После запуска `_Builder.bat` (или `./_Builder.sh`) пакеты автоматически подхватятся контейнером Image Builder и будут установлены в итоговую прошивку.

**Но тут начинаются проблемы...**

---

## 3. Реальная боль: почему пакеты не ставятся

### 3.1. Проблема №1: Несоответствие имён (баг `luci-i18n-podkop-ru`)

**Ситуация (из issue #44):** Пользователь скачал `luci-i18n-podkop-ru-0.260124.35205.apk`, но внутри пакета реальная версия — `0.7.14-r1`. Image Builder сверяет имя файла с внутренними метаданными и **отказывается ставить** пакет с несовпадающим именем.

**Что происходит:**
```
Установлен: luci-i18n-podkop-ru = 0.7.14-r1
Имя файла:  luci-i18n-podkop-ru-0.260124.35205.apk  ← НЕ СОВПАДАЕТ
Результат:  ERROR — unable to satisfy dependencies
```

**Решение:** Переименовать файл в соответствие с внутренней версией пакета:
```
luci-i18n-podkop-ru-0.260124.35205.apk  →  luci-i18n-podkop-ru-0.7.14-r1.apk
```

Раньше это приходилось делать вручную, вытаскивая версию через `unzip -p ... .PKGINFO` или угадывая. Теперь за вас это делает **APK Scanner**.

---

### 3.2. Проблема №2: Архитектура `noarch` отклоняется

**Ситуация (из issue #44):** Пользователь с устройством `radxa_rock_5t` (arch `aarch64_generic`) попытался импортировать пакеты с архитектурой `noarch`. Билдер ошибочно считал это несовместимым и отклонял.

**Что такое `noarch` и `all`:** Это **универсальные** архитектуры — пакеты без машинного кода (скрипты, Lua-файлы LuCI, конфиги). Они работают на **любом** устройстве.

**Решение (v4.60+):** Билдер теперь пропускает `noarch` и `all` как универсальные архитектуры без ошибки.

---

## 4. APK Scanner: автоматическая валидация и исправление

Начиная с **v4.60** в билдер встроен **APK Scanner** — утилита, которая до сборки проверяет и исправляет ваши `.apk` файлы.

### Что он делает

1. **Извлекает метаданные** через Docker:
   ```bash
   docker run --rm alpine:latest apk adbdump -- /package.apk
   ```
   Это читает `.PKGINFO` внутри APK без unzip — единственный надёжный способ получить реальную версию и архитектуру.

2. **Сверяет имя файла** с внутренними метаданными (`name-version-release.arch.apk`).

3. **Проверяет архитектуру** пакета против архитектуры профиля.

4. **Предлагает переименовать**, если имя не совпадает (по умолчанию — да).

### Когда запускается

- **Автоматически:** перед `docker compose up` в режиме IB, если в `custom_packages/` есть `.apk` файлы.
- **Вручную:** через кнопку **`[S] APK Scanner`** в главном меню билдера.

### Пример работы

```
==========================================================
  APK СКАНЕР v1.0 [aarch64_generic] [RU]
==========================================================
Сканирую APK в... custom_packages/radxa_rock_5t_25122_ow_full

----------------------------------------------------------
ПАКЕТ: luci-i18n-podkop-ru-0.260124.35205.apk
  ⚠ НЕСООТВЕТСТВИЕ ИМЕНИ
  Имя файла   : luci-i18n-podkop-ru-0.260124.35205.apk
  Внутреннее  : luci-i18n-podkop-ru-0.7.14-r1
  Переименовать? [Y/n]: Y
  ✓ Переименован в luci-i18n-podkop-ru-0.7.14-r1.apk
----------------------------------------------------------

----------------------------------------------------------
ПАКЕТ: luci-theme-argon-2.4.3-r20250722.apk
  ✓ Имя соответствует
  Архитектура: noarch (Универсальная) — OK
----------------------------------------------------------

==========================================================
  ГОТОВО: проверено 7, переименовано 1, предупреждений 0
==========================================================
```

### Язык сканера

Сканер используете тот язык, что указан в профиле (`SYS_LANG=RU` или `SYS_LANG=EN`). При ручном запуске язык определяется автоматически по настройкам билдера.

## 5. Пошаговая инструкция: добавляем пакеты в IB

### Шаг 1: Создайте папку профиля

```bash
mkdir -p custom_packages/my_profile
```

### Шаг 2: Скопируйте APK-файлы

Положите все нужные `.apk` в эту папку. Неважно, как они называются — сканер исправит.

### Шаг 3: Запустите билдер

```bash
_Builder.bat     # Windows
./_Builder.sh    # Linux
```

Выберите профиль и нажмите **`B`** (Build). Сканер автоматически запустится перед сборкой.

### Шаг 4 (опционально): Проверьте вручную

Если хотите провалидировать пакеты без запуска сборки — нажмите **`S`** в главном меню.

## 6. Частые вопросы (FAQ)

**В: Обязательно ли переименовывать файлы?**  
Да, Image Builder сверяет имя файла с внутренними метаданными. Сканер делает это за вас автоматически.

**В: Что делать, если сканер не может прочитать метаданные?**  
Ошибка `Parse Fail` означает, что `apk adbdump` не смог прочитать `.PKGINFO`. Убедитесь, что файл — корректный APK (не переименованный `.ipk` или `.deb`). Docker-образ `alpine:latest` должен быть доступен.

**В: Пакет `noarch` — это точно безопасно?**  
Да. `noarch`/`all` означают, что пакет не содержит машинного кода — это скрипты, конфиги, LuCI-интерфейсы. Они совместимы с любой архитектурой.

**В: Можно ли отключить автосканер?**  
Автосканер запускается только в режиме IB и только при наличии `.apk` файлов. Он не замедляет сборку — проверка занимает 2-5 секунд на пакет.

**В: Сканер сказал «Arch MISMATCH» для пакета с моей архитектурой**  
Это **предупреждение**, а не ошибка. Пакет будет проверен и добавлен, если вы подтвердите. Сканер лишь информирует, что архитектура пакета отличается от целевой (например, пакет `x86_64`, а профиль `aarch64_generic`).

**В: Где хранится образ `alpine:latest`, его можно удалить?**  
`alpine:latest` — служебный образ для сканирования. **Не удаляйте** его — он нужен для работы сканера. Занимает ~7 МБ.

## 7. Под капотом: как работает сканер

```
┌──────────────┐     docker run       ┌──────────────────┐
│  custom_     │  alpine:latest       │  apk adbdump     │
│  packages/   │ ──────────────────►  │  (читает .PKGINFO)│
│  *.apk       │                      │  name, version,  │
└──────────────┘                      │  release, arch   │
                                      └────────┬─────────┘
                                               │
                                        возвращает JSON
                                               │
                                               ▼
┌──────────────────────────────────────────────────────┐
│  СРАВНЕНИЕ                                           │
│  filename_name == metadata_name ✓                    │
│  filename_version == metadata_version ✓              │
│  filename_arch  == profile_arch   ✓ (или noarch)     │
│                                                      │
│  Если не совпадает → предложение переименовать       │
│  Если arch ≠ profile и ≠ noarch → предупреждение     │
│  exit 0 = всё хорошо, exit 1 = есть проблемы         │
└──────────────────────────────────────────────────────┘
```
