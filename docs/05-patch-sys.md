# 🛠 Руководство: Система патчей исходного кода (Source Patching)

## 1. Введение

В версии **4.33** внедрена новая система модификации исходного кода OpenWrt. Она разработана для решения одной из самых сложных задач при сборке прошивок: **как изменить исходный код проекта, не теряя изменений при пересоздании контейнера?**

Ранее для этого приходилось писать сложные скрипты в `hooks.sh` или править файлы внутри Docker вручную. Теперь процесс стал прозрачным и автоматизированным.

### Для чего это нужно?
*   **Модификация сборки:** Изменение `Makefile` для включения скрытых опций компиляции.
*   **Поддержка железа:** Добавление `.dts` файлов для новых устройств или патчей ядра.
*   **Исправление ошибок:** Применение `.patch` файлов к пакетам или системе.
*   **Кастомизация:** Подмена дефолтных конфигов внутри дерева исходников.

---

## 2. Ключевая концепция: "Зеркальный оверлей"

Система работает по принципу **зеркалирования**. Директория `custom_patches/<PROFILE_ID>/` на вашем компьютере — это проекция корневой папки исходников OpenWrt (`/home/build/openwrt/`) внутри контейнера.

> **Правило:** Любой файл, помещенный в `custom_patches`, будет скопирован в дерево исходников с сохранением структуры папок, заменяя оригинальные файлы.

### 🛡 Автоматическое исправление CRLF (Windows Fix)
Одной из главных проблем при разработке на Windows являются окончания строк (CRLF), которые ломают сборку в Linux.
**Наша система решает это автоматически:**
Перед применением патчей специальная утилита `dos2unix` рекурсивно обрабатывает все ваши файлы, преобразуя их в безопасный Unix-формат (LF). Вы можете редактировать файлы в "Блокноте" и не бояться ошибок компиляции.

---

## 3. Как это работает (Lifecycle)

1.  **Подготовка:** Вы создаете структуру папок внутри `custom_patches/<PROFILE_ID>/`, повторяющую путь к целевому файлу.
2.  **Обнаружение:** При запуске `_Builder.bat` проверяет наличие патчей. Если они есть, в меню рядом с профилем загорается индикатор **X** (Patches).
3.  **Изоляция:** Папка с патчами монтируется в Docker-контейнер в безопасную зону (`/patches`).
4.  **Применение:**
    *   Скрипт конвертирует файлы в Unix-формат.
    *   Утилита `rsync` накладывает ваши файлы поверх исходного кода OpenWrt.
5.  **Сборка:** Компиляция (`make`) запускается на уже модифицированном коде.

## 4. Практические примеры

### Пример №1: Подмена файла (Makefile)
**Задача:** Изменить опции компиляции пакета `hostapd`.
**Метод:** Полная замена файла.

1.  Путь к файлу в OpenWrt: `package/network/services/hostapd/Makefile`
2.  Создайте такую же структуру на диске:
    ```text
    custom_patches/
    └── my_profile/               <-- Имя вашего профиля
        └── package/
            └── network/
                └── services/
                    └── hostapd/
                        └── Makefile  <-- Ваш измененный файл
    ```
**Результат:** При сборке оригинальный Makefile будет перезаписан вашей версией.

---

### Пример №2: Нативный патчинг ядра (Рекомендуемый метод) 🔥
**Задача:** Добавить поддержку нового чипа Flash-памяти в ядро Linux.
**Метод:** Использование встроенной системы сборки OpenWrt (без скриптов!).

OpenWrt автоматически ищет файлы `.patch` в папках `patches-x.x` внутри `target/linux`. Нам просто нужно "подбросить" туда файл.

1.  Узнаем архитектуру (например, `ramips`) и версию ядра (например, `5.10`).
2.  Целевой путь в исходниках: `target/linux/ramips/patches-5.10/`.
3.  Создаем структуру:
    ```text
    custom_patches/
    └── my_profile/
        └── target/
            └── linux/
                └── ramips/
                    └── patches-5.10/
                        └── 999-add-new-flash.patch
    ```
**Результат:** Система сборки OpenWrt при подготовке ядра увидит этот файл и применит его стандартными средствами (`quilt`/`patch`).

---

### Пример №3: Применение патча через Hooks (Сложный сценарий)
**Задача:** Применить сложный патч к корню системы, который требует особой логики, или если вы не хотите соблюдать структуру папок OpenWrt.

1.  Положите патч в корень папки патчей профиля:
    ```text
    custom_patches/
    └── my_profile/
        └── 001-complex-fix.patch
    ```
2.  При старте сборки этот файл окажется в корне исходников OpenWrt: `/home/build/openwrt/001-complex-fix.patch`.
3.  Используйте `custom_files/my_profile/hooks.sh` для его применения:
    ```bash
    #!/bin/bash
    # hooks.sh выполняется в корне /home/build/openwrt
    echo "Applying manual patch..."
    patch -p1 < 001-complex-fix.patch
    ```

## 5. Часто задаваемые вопросы (FAQ)

**В: В чем разница между `custom_files` и `custom_patches`?**
*   **`custom_files`**: Файлы, которые попадут в **готовую прошивку** (например, `/etc/config/network` на роутере). Они накладываются в самом конце сборки.
*   **`custom_patches`**: Файлы, которые меняют **исходный код** OpenWrt (например, исходники C/C++, Makefiles). Они накладываются *перед* компиляцией.

**В: Нужно ли мне делать `make clean` после изменения патчей?**
*   Да. Если вы изменили патчи ядра или ключевых пакетов, рекомендуется выполнить **Soft Clean** (Меню -> Очистка -> Soft Clean), чтобы OpenWrt пересобрал измененные компоненты.

**В: Как назвать патч?**
*   Для Примера №2 (нативный патчинг) имена важны. OpenWrt применяет их в алфавитном порядке.
    *   `0xx-` — патчи разработчиков OpenWrt.
    *   `9xx-` — пользовательские патчи (рекомендуется использовать префикс, например `999-my-fix.patch`, чтобы он применялся последним).

**В: При сборке с патчами появляется ошибка `mkdir: cannot create directory '/tmp_patches': Permission denied`. Что делать?**
*   Эта ошибка исправлена в новых версиях билдера (начиная с версии, где применением патчей занялся корректный шаг с правами доступа). Обновите билдер до актуальной версии с [Releases](https://github.com/iqubik/routerFW/releases) или из репозитория.
